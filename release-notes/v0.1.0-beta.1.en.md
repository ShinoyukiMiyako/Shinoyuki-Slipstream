# v0.1.0-beta.1 — First public test build (Forge 1.20.1 network-layer traffic optimization)

Slipstream is the network branch of the Shinoyuki optimization series: BetterAutoSave is for save/load IO, Slipstream is for outbound network bandwidth. This is the first public build, shipping a full outbound-traffic telemetry suite plus fall-back-capable bandwidth optimizations, all verified on an internal production server. The series follows one priority order: data safety > stability > compatibility > performance — any technique that would trade data fidelity, server stability, or client compatibility for bandwidth is off the table.

## What this build does

- Outbound traffic telemetry: server-side clientbound traffic analysis, measurement only, no protocol changes. Per packet type: count, pre-compression / on-wire bytes, compression ratio, outbound share; per player: a heatmap breakdown; chunk-broadcast redundancy (how often the same chunk is re-serialized for multiple players). Commands `/slipstream report | reset | capture`; snapshots and raw captures land in `logs/slipstream/`
- zstd on-wire compression: negotiated per connection, single-sided fall-back. If the peer does not have or has not enabled Slipstream, it falls back to vanilla zlib automatically; vanilla clients connect as usual with byte-identical wire output. Chunk packets see ~15-35% on-wire byte reduction (default level 3, adjustable 1-19)
- Large packet support: raises the vanilla packet size cap so map data, vehicle/structure NBT, blueprint pastes and other large packets no longer crash or get rejected — a built-in replacement for XLPackets / PacketFixer
- serialize-once broadcast dedup: within a single sync broadcast each chunk packet is compressed once and reused for the remaining receivers, skipping redundant Deflater work
- Small-packet aggregation (combat scenarios): coalesces bursts of tiny entity-state packets (motion / move / head-rotate) into one batch within a window, eliminating per-packet framing overhead and handing the whole batch to zstd. Internal production-server dense-combat testing shows ~one-third outbound reduction. Window defaults to 20ms
- L2 entity-field delta (combat scenarios): re-encodes high-frequency entity-state packets per-entity per-field as zigzag-varint deltas, lossless. Stacks on top of aggregation; internal testing shows a further ~18% entity-traffic reduction
- requireClient strict dual-side: when enabled, the server rejects at login any client that has not enabled all of its negotiated optimizations, letting large servers guarantee every player runs the optimizations. Off by default

## Safety and compatibility

- All processing runs on the netty event-loop, never touching the server main (tick) thread — no added MSPT, no TPS impact
- All protocol rewrites are lossless: the entity-field delta has a byte-exact round-trip unit-test gate; inbound zstd frames have decompression-bomb size guards
- Single-sided fall-back at zero compatibility cost: vanilla and clients without the mod connect as usual, with every negotiated optimization falling back to vanilla behavior for them
- Negotiated features (zstd / aggregation / L2) only take effect when both ends have the mod installed and the same option enabled; their capability channels register at startup, so toggling them requires a server (and client) restart

## Why it is a beta

- First public build: core features are verified in internal production-server testing, but the public multi-server sample is still small
- Negotiated and combat-scenario optimizations (aggregation / L2) ship conservatively off by default, with only zstd on by default; steady-state behavior is still under observation

## Configuration and commands

- Full options and commands are in the README. Key switches: `zstdEnabled` (default on), `aggregateEnabled` (default off), `l2Enabled` (default off), `requireClient` (default off)
- Config file `config/Shinoyuki-Optimize/slipstream/common.toml`

## Upgrading and installation

- Drop-in: place it in `mods/`; a single side already gets telemetry and zstd single-sided fall-back behavior
- Negotiated features (zstd / aggregation / L2 / large packets) require both ends to have the mod installed and enabled; after toggling these switches both ends need a restart
